# Nekote AI 統合失敗の分析 | 2025-11-16

## 1. 背景：中核となる設計原則

AI 統合の初期試行を分析する前に、我々が従うべき必須のルールセット（`PLAYBOOK.md`）を再確認することが不可欠である。この文書は、すべての設計の基礎となる「単一の真実の源 (Single Source of Truth)」である。

AI アシスタントへの指示は、以下の 2 つの厳格な原則に基づいている：

1.  **関心の分離 (SoC - Separation of Concerns):**
    これは最も重要な原則である。すべてのクラス、メソッド、モジュールは、単一の明確に定義された責務を持たなければならない。
2.  **ドメイン・ファースト (アンチ・コラプション・レイヤー):**
    アーキテクチャは、内部の**ドメイン（ビジネスロジック）**と外部の**インフラストラクチャ（データソース）**間の厳格な分離に基づいている。これは、「アンチ・コラプション・レイヤー (ACL)」によって達成される。

この ACL の設計は、以下の 3 つのコンポーネントを**厳格に**区別することを要求する：

* **ドメインモデル (Domain Model):**
    純粋なビジネス概念（例: `NekoteChatMessage`）。外部システムの属性（`[JsonPropertyName]` など）を**一切含まない**、クリーンな POCO でなければならない。
* **DTO (Data Transfer Object):**
    外部コントラクト（例: OpenAI の API JSON）のデータ構造と**完全に一致**すること*だけ*が責務のクラス。シリアライズ属性を持つことが許される唯一の場所である。
* **マッパー (Mapper):**
    *特定の DTO* と *ドメインモデル* の間を翻訳する「橋渡し」。DTO とドメインモデルの両方を知る唯一の場所である。

## 2. 初期の失敗アプローチ：「共通ドメインモデル」という幻想

初期の AI 統合アプローチ（AI の提案に基づく）は、上記の中核原則、特に「ドメインモデルの純粋性」に違反した。

提案された戦略は「OpenAI の仕様をベースにした共通ドメインモデルを作成し、他の AI（Gemini など）のデータをそこへ変換すれば管理可能である」というものだった。

このアプローチは、以下の根本的な欠陥を抱えていた：

1.  **「共通ドメインモデル」の作成:**
    OpenAI の仕様（メッセージ構造、ロール名など）に強く結合した「ドメインモデル」を作成した。
2.  **原則違反:**
    この時点で、このモデルはもはや「ドメインモデル」ではない。それは「特定のベンダー仕様に汚染された、インフラストラクチャ層の平均化された DTO」である。これは、ドメインモデルが**純粋**でなければならず、外部システムの属性を**一切含んではならない**という `PLAYBOOK.md` の規則 (セクション 3.2) への明確な違反である。

## 3. 発生した症状とアーキテクチャの破綻

この「汚染されたドメイン」アプローチは、予測可能ないくつかの破綻を引き起こした。

### A. 関心の分離 (SoC) の崩壊

最も深刻な症状は、コンポーネント間の責務が曖昧になったことである。

* **マッパーがバリデーターになる:**
    本来「A を B にマップする」だけが責務であるはずの `Mapper` が、Gemini から来たデータが「共通（OpenAI ベースの）モデル」に適合するかを検証する、複雑なバリデーションロジックを持つことになった。これは `Mapper` と `Validator` の責務を混同しており、SoC 原則に違反する。
* **ドメインモデルが DTO になる:**
    「共通ドメインモデル」は、OpenAI 以外のプロバイダ（Gemini, Anthropic）から見れば、単なるもう一つの「変換先 DTO」でしかない。これにより、`DTO -> DTO` という不要なマッピング層が生まれ、アーキテクチャ全体が複雑化した。

### B. 不要なモデルの増殖

このアプローチは、「すべてにドメインモデルが必要である」という誤った前提を生み出した。

例えば、翻訳や要約のような機能は、ドメイン層では「文字列を受け取り、文字列を返す」単純なインターフェース契約（例: `ITranslationService.TranslateAsync(string text)`) として表現されるべきである。

しかし、失敗したアプローチでは、これらの単純な操作に対しても `TranslationRequestModel` や `TranslationResponseModel` のような、OpenAI の仕様に引きずられた不要な「ドメインモデル」が作成された。

### C. 保守性の爆発（予測）

このアプローチが最終的に破綻した最大の理由である。

OpenAI の仕様は `content` が文字列または配列になり得る。Gemini や Anthropic の仕様は、ロールの扱い（例: `system` ロールの有無）やメッセージの構造が根本的に異なる。

「共通モデル」を維持しようとすると、`Mapper` や `Model` 自身が、以下のような**保守不可能なロジック**を持たなければならなくなる：

```csharp
// 悪い例：汚染されたドメインモデルまたはマッパー
if (provider == "openai") {
    // OpenAI の content 配列を処理
} else if (provider == "gemini") {
    // Gemini の ... を処理
} else if (provider == "anthropic") {
    // Anthropic の system プロンプトを処理
}
```

これは `Nekote.Core` が目指す「プロバイダの透過的な切り替え」 と正反対であり、ベンダーロックインをコードベースの中心に招き入れる行為である。

## 4. 結論：アンチ・コラプション・レイヤーの厳格な適用へ

初期の失敗は、「ドメイン・ファースト」と「アンチ・コラプション・レイヤー (ACL)」の原則 を AI の提案によって曲げてしまったことに起因する。

AI が「管理可能だ」と提案したとしても、それがアーキテクチャの**中核原則（特に SoC と ACL）**に違反する場合、その提案は拒否しなければならない。

教訓は明確である：**外部仕様（インフラ）が内部仕様（ドメイン）を定義してはならない。**

次のステップ（`nekote-ai-new-plan-2025-11-16.md` で詳述）は、この失敗を正し、`PLAYBOOK.md` および `nekote-ai-integration-strategy` の原則に厳格に従った、クリーンなアーキテクチャを再構築することである。
